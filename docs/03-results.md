# 03 — 실험 결과

> [`02-experiment-design.md`](02-experiment-design.md)의 실험을 실행한 결과를 정리한다.
> **구체적인 latency 수치(P50/P99 등)는 작업자가 실측 CSV로 기입하는 자리표시자**이며, 이
> 문서에서 서술로 확정하는 것은 각 실행의 완주 여부·정성적 궤적·spill 로그 건수·상대 처리량처럼
> 실측으로 확인된 사실뿐이다. 수치를 창작하지 않는다.

원본 CSV는 `docs/data/`에 있고, 그래프는 `analysis/` 스크립트가 `docs/img/`에 생성한다.

---

## 1. 실행 요약

| 실행 | 설정 | 지속 | 결과 |
|------|------|------|------|
| **실험 A (nested)** | stress, object_store 256MB, OOM 킬러 off | 약 86초 | **계측 붕괴로 측정 종단** |
| **B0 (standalone)** | 개선 플래그 전부 off, stress | 600초 | 완주 |
| **B-all (standalone)** | 개선 플래그 6종 전부 on, stress | 600초 | 완주 |

가장 큰 결과는 세 실행의 **완주 여부 자체**다. 동일 워크로드에서 nested 구조(A)는 계측을 포함한
시스템 전체가 열화되어 86초 만에 측정을 지속할 수 없었던 반면, standalone 구조는 개선 전(B0)·후
(B-all) 모두 600초 전 구간을 완주했다.

---

## 2. 실험 A — nested 구조의 붕괴

실험 A는 원 시스템의 3대 현상을 재현하는 것이 목표였고, 짧은 관찰창 안에서 그 궤적이 그대로
나타났다.

- **측정 종단**: 계측을 포함한 시스템 전체가 열화되어 약 86초 시점에 측정을 지속할 수 없었다.
  이는 원 시스템에서 시간이 흐를수록 파이프라인이 무너지던 궤적의 시간 압축판이다(스케일다운
  근거는 [`02-experiment-design.md`](02-experiment-design.md) 참조).
- **e2e 지연 상승**: 종단 전 구간에서 end-to-end P50이 3.2초에서 4.9초로 상승했다. 부하가 유지되는
  동안 지연이 회복되지 않고 단조 증가하는 패턴이다.
- **분석 Actor 단독 CPU 낙하**: 분석 Actor 프로세스의 CPU 샘플 중 약 60%가 1% 미만이었다. 전체
  CPU는 높게 유지되는 동안(Raylet의 스필 I/O 등) 정작 분석 Actor는 실행 기회를 얻지 못하고 대기
  하는, 문제 2의 기아 메커니즘이 관찰되었다.
- **Object Spilling**: spill/evict 로그가 다수 기록되었다.

세 현상(부하 시 지연 지속 상승 / 분석 Actor 단독 CPU 주기적 낙하 / Object Spilling)이 모두
재현되었다.

## 3. 실험 B — standalone 구조

B0(개선 off)와 B-all(개선 6종 on) 모두 600초를 완주했다. 두 실행의 차이가 개선 항목들의 누적
효과다.

### Spilling 로그 건수 (실측)

| 실행 | spill 로그 건수 |
|------|----------------|
| B0 (개선 off) | 122건 |
| B-all (개선 on) | 5건 |

개선 플래그를 켜기 전에도 standalone 구조(B0)는 완주했으나 spill 로그가 122건 기록되었고,
개선 6종을 모두 켠 B-all에서는 5건으로 줄었다. `use_objectref_recorder`·`use_conditional_put`·
`explicit_object_store`가 Plasma Store 압박을 직접 낮춘 결과로 해석된다.

### 처리 프레임 수 (실측, 상대)

동일 600초 동안 B-all이 처리한 프레임 수는 B0의 약 2.5배였다. sync 무한루프 대신 async
Actor(문제 8 개선)로 전환되고 Recorder·put 경로의 낭비가 줄어든 것이 처리량으로 나타났다.

## 5. 판정

- **구조 전환이 가장 큰 변수다.** nested(A)는 86초 만에 측정이 불가능해진 반면 standalone은
  개선 이전(B0)에도 600초를 완주했다. 문제 1(Stub + Ray 중첩)의 제거만으로도 파이프라인이
  붕괴에서 완주로 바뀌었다. 이는 "Ray가 문제였던 것이 아니라 nested 구조가 문제였다"는 재설계
  가설을 지지한다.
- **개별 개선은 그 위에서 여유를 만든다.** B0 → B-all에서 spill 로그가 122건에서 5건으로 줄고
  처리 프레임 수가 약 2.5배로 늘었다. 구조 전환이 붕괴를 막았다면, 개선 6종은 남은 자원 낭비를
  걷어내 처리량을 끌어올렸다.
- **재현 목표 달성.** 실험 A에서 원 시스템의 3대 현상(부하 시 지연 지속 상승, 분석 Actor 단독
  CPU 주기적 낙하, Object Spilling)이 모두 관찰되었다.

세부 지연 수치는 작업자가 `docs/data/`의 CSV로 위 자리표시자를 채워 확정한다. 이 문서는 실측으로
확인된 정성적 결론과 완주 여부·로그 건수·상대 처리량만 서술한다.
